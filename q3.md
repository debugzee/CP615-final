# Observations

- `Employee.empID` is a **primary key**
- `Department.deptID` is a **primary key**
- `Office.oID` is a **primary key**
- `Task(empID, task)` is a **composite primary key**
- `Employee.dept` is a **foreign key** to `Department.deptID`
- `Employee.office` is a **foreign key** to `Office.oID`
- `Department.mainOffice` is a **foreign key** to `Office.oID`
- `Employee` participates totally in `Works_for`, so every employee must belong to a department
- `Employee` participates totally in `Hosted_in`, so every employee must be assigned an office
- `Department` participates totally in `Operates_from`, so every department must have a main office
- The `Hosted_in` and `Operates_from` relationships are treated as **1:1**, so:
  - one office cannot host more than one employee

## Answers

### (1) Insert into `Employee`: `('E007', 'John Smith', 30000, 'BL10', null)`

**Yes**, this violates a constraint.

Reason: `dept` is `null`, but every employee must work for exactly one department. This violates the **participation/cardinality constraint** of `Works_for`.

### (2) Insert into `Employee`: `('E008', 'Sara Lee', 25000, 'BL07', 'D001')`

**Yes**, this violates a constraint.

Reason: `BL07` is already assigned to employee `E004`. Since `Hosted_in` is 1:1, one office cannot host more than one employee. This violates the **cardinality constraint** of `Hosted_in`.

### (3) Insert into `Employee`: `('E009', 'Hanna Shon', 45000, 'BL10', 'D001')`

**No**, this action causes no violation.

Reason:

- `E009` is a new employee ID
- `BL10` exists in `Office`
- `BL10` is not currently assigned to another employee
- `D001` exists in `Department`

### (4) Insert into `Employee`: `('E007', 'Salma Hayek', 45000, null, 'D001')`

**Yes**, this violates a constraint.

Reason: `office` is `null`, but every employee must be hosted in exactly one office. This violates the **participation/cardinality constraint** of `Hosted_in`.

### (5) Insert into `Employee`: `('E008', 'Maya Mi', 55000, 'BL01', 'D001')`

**No**, this action causes no violation.

Reason:

- `E008` is a new employee ID
- `BL01` exists
- `D001` exists

Using `BL01` as an employee office does not clearly violate any constraint shown in the figure, even though `BL01` is also the main office of a department.

### (6) Insert into `Employee`: `('E005', 'David Johnson', 55000, 'BL04', 'D001')`

**Yes**, this violates more than one constraint.

Reasons:

1. `E005` already exists, so this violates the **primary key constraint** on `Employee.empID`.
2. `BL04` is already assigned to employee `E001`, so this violates the **1:1 cardinality constraint** of `Hosted_in`.

### (7) Insert into `Employee`: `('E055', 'Javed Font', 65000, 'BL10', 'D004')`

**Yes**, this violates a constraint.

Reason: `D004` does not exist in `Department`, so this violates the **foreign key constraint** on `Employee.dept`.

### (8) Insert into `Department`: `('D04', 'HR', 'St. George', null)`

**Yes**, this violates a constraint.

Reason: `mainOffice` is `null`, but every department must operate from exactly one office. This violates the **participation/cardinality constraint** of `Operates_from`.

### (9) Insert into `Department`: `('D005', 'Production', 'St. George', 'BL03')`

**Yes**, this violates a constraint.

Reason: `BL03` is already the main office of department `D003`. Since `Operates_from` is 1:1, one office cannot be the main office of more than one department. This violates the **cardinality constraint** of `Operates_from`.

### (10) Insert into `Department`: `('D006', 'PR', 'St. George', 'BL04')`

**No**, this action causes no violation.

Reason:

- `D006` is a new department ID
- `BL04` exists in `Office`
- `BL04` is not currently used as a `mainOffice` by any department

### (11) Insert into `Task`: `('E001', 'Modeling')`

**No**, this action causes no violation.

Reason:

- `E001` exists in `Employee`
- `('E001', 'Modeling')` is a new `(empID, task)` pair

An employee can have multiple tasks.

### (12) Insert into `Task`: `('E006', 'Financial Reporting')`

**No**, this action causes no violation.

Reason:

- `E006` exists in `Employee`
- `('E006', 'Financial Reporting')` is a new `(empID, task)` pair

### (13) Insert into `Task`: `(null, 'Office Management')`

**Yes**, this violates a constraint.

Reason: `empID` is part of the composite primary key of `Task`, so it cannot be `null`. This violates **entity integrity**.

### (14) Insert into `Task`: `('E006', null)`

**Yes**, this violates a constraint.

Reason: `task` is part of the composite primary key of `Task`, so it cannot be `null`. This violates **entity integrity**.

### (15) Insert into `Task`: `('E007', 'Office supplies')`

**Yes**, this violates a constraint.

Reason: `E007` does not exist in the current `Employee` table, so this violates the **foreign key constraint** on `Task.empID`.

### (16) Delete from `Office`: `('BL01', 100)`

**Yes**, this violates a constraint.

Reason: `BL01` is referenced by department `D001` as its `mainOffice`. Deleting it would violate **referential integrity / foreign key constraint**.

### (17) Delete from `Office`: `('BL10', 170)`

**No**, this action causes no violation.

Reason: `BL10` is not referenced by any employee or department in the current database state.

### (18) Delete from `Task`: `('E001', 'Data Analysis')`

**No**, this action causes no violation.

Reason: No other table depends on this task row, and there is no constraint requiring an employee to always have at least one task.

### (19) Delete from `Department`: `('D001', 'Research', 'St. George', 'BL01')`

**Yes**, this violates a constraint.

Reason: employees `E001` and `E002` reference `D001` in `Employee.dept`. Deleting `D001` would violate **referential integrity / foreign key constraint**.

### (20) Delete from `Employee`: `('E006', 'Li Wei', 75000, 'BL09', 'D003')`

**No**, this action causes no violation.

Reason: there is no task for `E006` in the current `Task` table, so deleting this employee does not violate any foreign key constraint.
